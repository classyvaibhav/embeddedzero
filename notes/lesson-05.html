What Is an Embedded System?

Before you write code, understand the machine. Microcontrollers vs microprocessors. Memory. Boot process. The toolchain that turns your code into running firmware.

By the end of this lesson, you will:
  • Define an embedded system and name 5 in your own life
  • Explain microcontroller vs microprocessor and know which fits where
  • Understand the memory map of an MCU (Flash, SRAM, EEPROM)
  • Describe what happens from power-on to your code running

1The Definition

An embedded system is a dedicated computer built into a larger product, doing one specific job. Not a general-purpose PC. It usually has:

  • Fixed function — does one thing well (control a motor, read a sensor, drive a display)
  • Real-time behaviour — responds in milliseconds or microseconds
  • Limited resources — tiny memory compared to a laptop
  • Direct hardware interaction — talks to sensors, motors, LEDs, not keyboards
  • Often low-power — might run on a battery for months or years

You're surrounded by embedded systems right now. Let's count:

Embedded systems are literally everywhere. 10+ billion MCUs ship per year.

3Memory in an MCU

Inside every MCU are different memory regions, each with a specific purpose. Understand them now — it'll save you hours of confusion later.

Memory TypeVolatile?Size (ATmega328P)What Lives Here
FlashNo (survives power-off)32 KBYour compiled firmware, constants
SRAMYes (wiped on reset)2 KBVariables, stack, heap at runtime
EEPROMNo1 KBUser settings, calibration data
Registers(volatile)TinyPeripheral config (GPIO, UART, timers)
Flash holds your firmware; SRAM holds live variables; registers control the peripherals.
// Where does each of these variables live?

const int ledPin = 13;      // "const" → Flash (read-only)
int counter = 0;              // SRAM (variable, runtime)
char buffer[32];             // SRAM (array)

void setup() {
  pinMode(ledPin, OUTPUT);   // writes GPIO register
  EEPROM.write(0, 42);       // writes EEPROM (survives reset)
}

4What Happens on Power-On?

When you plug in your Arduino, a specific sequence happens before your code runs:

A full cold-boot takes ~1 second. Compare to your laptop, which takes 30 seconds to boot Windows.
  1. Power stabilises — regulator brings Vcc up cleanly; decoupling caps fill up.
  2. Reset released — /RESET pin goes HIGH via pull-up; CPU starts executing instructions.
  3. Clock starts — crystal oscillator stabilises; CPU clock is now valid.
  4. Bootloader runs — a tiny program in Flash checks if a new firmware upload is pending over USB/UART. If not, it jumps to your firmware.
  5. C startup code runs — initialises variables, sets up stack.
  6. setup() runs — your initialisation code.
  7. loop() runs forever — until you power off or reset.

5The Toolchain — Code to Firmware

When you click "Upload" in Arduino IDE, a whole pipeline runs:

Cross-compilation: your laptop compiles code for a different CPU (the AVR/Xtensa inside the MCU).

This is called cross-compilation: your laptop is an x86 or ARM machine, but the MCU has a different CPU (AVR for Arduino Uno, Xtensa for ESP32). The compiler produces machine code for the target CPU, not your laptop.

💡 Why You Can't Run a .exe on Arduino A .exe file is compiled for an x86 Windows system. Arduino's AVR doesn't speak x86. So you can't just copy a program from your PC to the Arduino — you need to cross-compile specifically for the AVR target. Arduino IDE does this automatically.

🔍 Check Yourself

Name one embedded system in your kitchen.
Microwave, fridge controller, induction cooker, rice cooker, electric kettle with temp control, gas leak detector — all embedded!
Is ESP32 a microcontroller or microprocessor?
Microcontroller (sometimes called SoC because it has Wi-Fi/BT built in). Everything is on the chip — CPU, RAM, Flash, peripherals.
Where does a const int x = 10; variable live?
Flash. The const keyword tells the compiler the value never changes, so it's stored in read-only Flash memory to save precious SRAM.
Why can't a typical MCU run Linux?
Linux needs at least ~64 MB of RAM. Most MCUs have <512 KB. Also Linux needs an MMU (memory management unit) that most MCUs lack.
What happens between power-on and setup()?
Power stabilises → reset released → clock starts → bootloader runs → C runtime initialises → setup(). Total ~1 second.

Key Takeaways

  • Embedded system = dedicated computer for a specific job, built into a product.
  • MCU = everything on one chip (Arduino, ESP32). MPU/SoC = needs external RAM, runs Linux (Raspberry Pi, phones).
  • MCU memory: Flash (your code), SRAM (variables), EEPROM (settings), Registers (peripheral control).
  • Boot sequence: power → reset → clock → bootloader → setup() → loop().
  • Arduino uses cross-compilation: compile on PC, run on MCU.
