Interrupts & Non-Blocking Code

Your code can only do one thing at a time — or can it? Use interrupts for instant response to events, and millis() to do multiple things seemingly at once.

By the end of this lesson, you will:
  • Attach an interrupt to a GPIO pin to detect events instantly
  • Know why you must NEVER put Serial.println() inside an ISR
  • Use millis() instead of delay() to do two tasks simultaneously

1The Problem with delay()

Look at this sketch. What's wrong with it?

void loop() {
  digitalWrite(LED, HIGH);
  delay(1000);               // ↪ CPU frozen here for 1 second
  digitalWrite(LED, LOW);
  delay(1000);               // ↪ CPU frozen here too

  if (digitalRead(BTN) == LOW) {
    Serial.println("Pressed!");
  }
}

The button check only runs for a brief moment every 2 seconds. If you press the button during a delay(), it's missed entirely.

delay() is a blocking function — the CPU can literally do nothing else while it waits. For simple sketches it's fine, but as soon as you want to do two things, it falls apart.

2Solution 1: millis() for Non-Blocking Intervals

millis() returns the number of milliseconds since the MCU started. It keeps running forever. We can use it to check "has 500 ms passed since I last did X?" without blocking:

const int LED = 13;
const int BTN = 4;
unsigned long lastBlink = 0;
bool ledState = false;

void setup() {
  pinMode(LED, OUTPUT);
  pinMode(BTN, INPUT_PULLUP);
  Serial.begin(115200);
}

void loop() {
  unsigned long now = millis();

  // Task 1: blink LED every 500 ms
  if (now - lastBlink >= 500) {
    lastBlink = now;
    ledState = !ledState;
    digitalWrite(LED, ledState);
  }

  // Task 2: check button as fast as possible
  if (digitalRead(BTN) == LOW) {
    Serial.println("Button pressed!");
    delay(50);  // simple debounce
  }
}

Now the LED blinks every 500 ms AND the button is responsive the whole time. The loop runs thousands of times per second, checking both.

🌎 Universal Pattern if (millis() - lastTime >= interval) { lastTime = millis(); /* do stuff */ } — memorise this. It's the foundation of all non-blocking code. You'll use it daily.

3Solution 2: Interrupts for Instant Response

Interrupts let hardware trigger code the moment an event happens — no polling needed. Common uses: a button press, a pulse from a sensor, a timer expiring.

const int BTN = 4;
volatile bool pressed = false;

void IRAM_ATTR onButton() {
  pressed = true;              // just set a flag — that's all!
}

void setup() {
  Serial.begin(115200);
  pinMode(BTN, INPUT_PULLUP);
  attachInterrupt(digitalPinToInterrupt(BTN), onButton, FALLING);
}

void loop() {
  if (pressed) {
    Serial.println("Button was pressed!");
    pressed = false;
  }
}

The function onButton() is called an ISR (Interrupt Service Routine). It runs the instant the button goes LOW — even if the main loop is in the middle of something else.

⚠ ISR Rules You Must Follow
  • Keep it SHORT. Set a flag, return. Don't do work inside.
  • No Serial.println() inside. No delay(). No I2C. It'll hang or crash.
  • Shared variables must be volatile. Tells the compiler "this can change unexpectedly".
  • On ESP32, use IRAM_ATTR. Keeps the function in fast RAM.

4Debouncing — Why Buttons Misbehave

Physical buttons don't cleanly go from "not pressed" to "pressed". Metal contacts bounce for 1–10 milliseconds. A single physical press can register as 3–5 electrical transitions.

Fix in software: ignore changes that happen within 50 ms of the last change.

volatile unsigned long lastTime = 0;

void IRAM_ATTR onButton() {
  unsigned long now = millis();
  if (now - lastTime > 50) {     // 50 ms since last interrupt?
    count++;
    lastTime = now;
  }
}

🔍 Check Yourself

Why avoid delay() in multi-task code?
It blocks the CPU. No other code runs during the delay. Use millis() pattern instead.
What does volatile mean?
Tells the compiler: "this variable can change unexpectedly, don't optimise reads away; always re-fetch from memory". Required for variables shared between ISR and main loop.
What should you NEVER do inside an ISR?
Long operations, Serial.print(), delay(), I2C calls, anything slow. The ISR must return quickly so the rest of the system keeps running.
Why do buttons sometimes register multiple times per press?
Mechanical bounce — metal contacts vibrate briefly on press/release. Fix with software debounce (ignore events within 50 ms).

Key Takeaways

  • delay() blocks the CPU. Fine for Blink; useless for real multi-tasking.
  • millis() gives non-blocking intervals. Pattern: if (now - last >= interval) { last = now; /* work */ }
  • Interrupts trigger code the instant an event happens. Use attachInterrupt(pin, func, mode).
  • ISR rules: short, no prints, volatile shared vars, IRAM_ATTR on ESP32.
  • Buttons bounce — always debounce in software (~50 ms threshold).
