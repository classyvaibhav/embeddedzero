What Is an Embedded System?

Session 22 · 45 min · Foundational
Goal: She can define an embedded system, give 5 examples, and explain why it's different from a general-purpose computer.

An embedded system is a dedicated computer built into a larger device to perform a specific task — not a general-purpose PC. It usually has:

  • Fixed function: does one thing well (regulate motor speed, read a sensor, drive a display).
  • Real-time constraints: responds within milliseconds or microseconds.
  • Limited resources: tiny RAM (kilobytes), tiny flash, often no OS or a lightweight RTOS.
  • Direct hardware interaction: talks to GPIOs, sensors, motors, not a user via keyboard.
  • Often battery-powered or low-power.

She's already surrounded by embedded systems — she just hasn't labelled them that way.

Standard architecture: sensors (input) → MCU (process) → actuators (output), all on a power supply.

Teach this way
  1. Ask her to list 5 devices around her that might have an embedded system inside. She'll list her phone, smart TV, AC, microwave, fitness band. All correct.
  2. Draw the block diagram on whiteboard: sensors → MCU → actuators.
  3. Discuss her Aimtron dashcam specifically. What are the inputs? (camera, GPS, accelerometer). What's the MCU doing? (video processing, event detection, cloud comms). What are the outputs? (flash LED, beep, upload video).
  4. Contrast with a laptop: general-purpose, runs many applications, has OS, keyboard/screen. Embedded runs one dedicated function.
  5. Explain why embedded matters: it's everywhere; 10 billion MCUs ship per year. Jobs are plentiful if you know the basics.
Common confusion
  • Confusing embedded with "the cloud" or "IoT". IoT is a subset of embedded — internet-connected embedded.
  • Thinking embedded = just the MCU chip. Embedded = the whole system: MCU + peripherals + sensors + actuators + power + firmware.
  • Assuming embedded always needs an OS. Most tiny MCUs run bare-metal "super-loop" code (like Arduino). Big embedded (her Aimtron) runs embedded Linux.
Verify understanding
  1. Name one embedded system in her kitchen. Microwave, fridge controller, induction cooker, rice cooker, etc.
  2. 3 differences between embedded and a laptop? Fixed function, limited resources, direct hardware interaction
  3. Is IoT = embedded? IoT is a subset — specifically networked embedded systems
T3.1

Microcontroller vs Microprocessor

Session 23 · 40 min · Foundational
Goal: She can distinguish MCU vs MPU, give an example of each, and explain when to choose which.

Microcontroller (MCU): Everything on one chip — CPU core, RAM, Flash (program memory), peripherals (UART, I2C, ADC, timers). Cheap, low-power, boots instantly, programmed with a single binary. Examples: ATmega328P (Arduino Uno), STM32F103, ESP32.

Microprocessor (MPU): Just the CPU core. Needs external RAM, Flash / disk, sometimes external peripheral chips. Runs a full OS (Linux, Android). More powerful, higher power draw. Examples: NXP i.MX8M Plus (Vaibhav's work), Raspberry Pi BCM2711, any laptop/phone SoC.

Teach this way
  1. Pull out the ESP32 DevKit. Point to the ESP32 chip — that's an MCU.
  2. If you have a Raspberry Pi around, pull that out. Show the BCM2711 chip and the separate DDR RAM chip + SD card slot. That's an MPU-based system.
  3. Tie to her world: Aimtron dashcam uses an MPU/SoC (your NXP i.MX8M Plus). Why? Because it runs video decode, AI inference, Linux — far more than an MCU can do.
  4. Tie to her target job: she's aiming at MCU-level roles (ESP32, STM32). The MPU / Yocto / embedded-Linux world is 1–2 years down the track.
  5. Edge case: ESP32 is weird — technically an MCU (everything on-chip) but has Wi-Fi + Bluetooth + dual CPU. Sometimes called an "SoC" (System-on-Chip). SoC is a fuzzy term.
Common confusion
  • Thinking Raspberry Pi is a microcontroller. It isn't — it's a Linux single-board computer.
  • Arduino being called a "processor". The Arduino Uno has an MCU (ATmega328P). The board is a dev platform.
  • Not realising that running Linux requires external RAM, which MCUs don't have.
Verify understanding
  1. Is ESP32 an MCU or MPU? MCU (sometimes called SoC); everything on-chip
  2. Why can't a typical MCU run Linux? Not enough RAM (typically <512 KB vs Linux needing ~64 MB+)
  3. Which to use for a washing machine controller? Drone video processing? MCU for washing machine, MPU for drone video
T3.2

Memory Types — Flash, RAM, EEPROM, Registers

Session 24 · 40 min · Foundational
Goal: She can explain the 4 memory regions in an MCU and where code, variables, and saved settings live.

Inside an MCU are several kinds of memory with different purposes:

  • Flash (program memory): Non-volatile. Holds the compiled firmware. ~KB to MB. Survives power-off. Only rewritten during programming (10,000+ erase cycles typical).
  • SRAM (working memory): Volatile. Holds variables, stack, heap at runtime. KB range. Wiped on reset.
  • EEPROM / Data Flash: Non-volatile, byte-writable. For user settings, calibration data. Lasts ~100,000 writes. (ESP32 emulates EEPROM in flash.)
  • Registers: Tiny but fast hardware registers inside the CPU and peripherals. Used directly for GPIO states, peripheral config. Arduino hides these; bare-metal C writes them directly.
Flash stores the firmware; SRAM holds live variables; registers control peripherals.
// Where do these variables live on an Arduino/ESP32?

const int ledPin = 13;      // "const" → Flash (read-only)
int counter = 0;             // SRAM (runtime variable)
char name[32];              // SRAM (array buffer)

void setup() {
  pinMode(ledPin, OUTPUT);  // writes GPIO peripheral register
  EEPROM.write(0, 42);      // writes EEPROM (emulated on ESP32)
}
Teach this way
  1. ATmega328P (Arduino Uno) spec: 32 KB Flash, 2 KB SRAM, 1 KB EEPROM. Read it off the datasheet. Concrete numbers.
  2. ESP32 spec: 4 MB Flash, 520 KB SRAM. No real EEPROM (emulated).
  3. Walk through the code above. Ask: "Where does each variable live?"
  4. Explain what happens on reset: SRAM clears, Flash keeps firmware, EEPROM keeps settings.
  5. Demonstrate: run a sketch that counts, store the count in EEPROM, power-cycle, count resumes.
Common confusion
  • Declaring big arrays / strings in RAM that don't fit. On Arduino Uno with 2 KB SRAM, a 1500-byte array leaves almost nothing for the stack → crash.
  • Expecting variables to survive reset. They don't — use EEPROM for that.
  • Excessive EEPROM writes — wear it out. Don't write every loop iteration; write on state change only.
Verify understanding
  1. Where does compiled firmware sit? Flash
  2. Where does a global int x = 5; live? SRAM
  3. ATmega328P has 2 KB SRAM. Can you declare char big[3000];? No — exceeds RAM
T3.3

The Boot Process — What Happens on Power-On

Session 25 · 35 min · Foundational
Goal: She can describe, in order, what happens from power-on to loop() running.

Every embedded system powers up in the same general sequence:

  1. Power stabilises. Regulator brings Vcc up cleanly. Decoupling caps fill.
  2. Reset released. Reset pin pulled high (via RC delay or external circuit). CPU starts executing.
  3. Clock starts. Crystal oscillator (or internal RC) stabilises. CPU clock valid.
  4. Bootloader runs (optional). On Arduino/ESP32, a small bootloader in Flash checks if a new firmware upload is pending (over UART/USB). If not, it jumps to user firmware.
  5. User firmware starts. C startup code initialises .bss (zero-fill), copies .data (initialised variables) from Flash to RAM, sets up stack pointer.
  6. main() / setup() runs. Your code finally begins.
  7. Main loop runs forever. On Arduino, loop() called repeatedly.

On an MPU running Linux (your Aimtron dashcam world): add U-Boot, kernel, init system, user apps — much longer chain. But for Arduino/ESP32, steps 1–7 is all.

Typical boot timeline for an Arduino/ESP32 — total ~1 second cold-start.
Teach this way
  1. Draw the timeline on whiteboard.
  2. Contrast with laptop boot: "Your laptop takes 30 seconds because it loads a full OS. This runs in a second because it's dedicated."
  3. Show the ESP32 programming dance: RTS/DTR signals from CP2102 pulse EN low and BOOT low simultaneously to enter the UART bootloader.
  4. Talk about fail-safe boot: if user firmware is corrupted, the bootloader often remains recoverable. Critical for field-upgradable devices.
  5. Tie to her world: Aimtron dashcam boots U-Boot → kernel → init → apps. Your domain.
Common confusion
  • Confusing "bootloader" with "firmware". Bootloader = small helper that can load firmware. Firmware = your code.
  • Assuming setup() runs instantly. There's ~1s of reset + clock + bootloader before setup() on a typical board.
  • Forgetting Flash → RAM copy of initialised variables. That's the reason your variables have the values you assigned at declaration time.
Verify understanding
  1. What's the first thing after power-on? Power rail stabilises, then /RESET released
  2. What runs before setup()? Bootloader + C runtime init
  3. Where does the bootloader live? Reserved area in Flash
T3.4

Toolchain — Compiler, Linker, Upload

Session 26 · 35 min · Foundational
Goal: She can explain what happens when she clicks "Upload" in Arduino IDE — source code to running firmware.

From her C code to a running LED blink, several steps happen:

  1. Preprocessor: expands #include, #define. Produces plain C.
  2. Compiler: converts C / C++ to machine instructions for the target CPU (e.g., Xtensa for ESP32, AVR for Arduino Uno, ARM for STM32). Output: object file (.o).
  3. Linker: stitches together object files + libraries, resolves addresses, produces a single binary (.elf / .bin / .hex).
  4. Uploader: sends the binary over USB/serial to the MCU's bootloader, which writes it to Flash.
  5. Reset: MCU restarts, now running the new firmware.

This is cross-compilation: the compiler runs on her laptop (x86/ARM) but produces code for a different CPU (AVR/Xtensa). That's why you can't just run a .exe on Arduino — the instruction set is different.

Arduino IDE hides these steps. You can see them in verbose output.
Teach this way
  1. Turn on verbose output in Arduino IDE (File → Preferences → Show verbose output).
  2. Upload Blink. Show her the avr-gcc / xtensa-gcc calls scrolling by.
  3. Point out: compile first, link second, upload third. Each step produces a file.
  4. Explain cross-compilation: her laptop is x86; MCU is AVR/Xtensa/ARM. Different CPU ISAs. Compiler has to target the MCU.
  5. Find the .hex file on disk (temporary folder); open in a hex editor. She sees the actual bytes that go into Flash.
  6. Tie to your world: on NXP i.MX8M Plus you run an ARM64 cross-compiler producing Linux executables. Same idea, bigger scale.
Common confusion
  • Thinking you can run a PC .exe on an MCU. Different CPU, different instruction set, no OS.
  • Conflating compile errors and link errors. Compile = "syntax wrong in your file". Link = "function called but not defined anywhere".
  • "It compiled but nothing happens on the board" — usually an upload problem (wrong port, wrong board, COM driver). Check each step.
Verify understanding
  1. What's cross-compilation? Compiling on one CPU for a different CPU
  2. What's the linker's job? Combine object files, resolve addresses, produce final binary
  3. What physical medium receives the uploaded binary? MCU's Flash
T3.5

Arduino Sketch Structure — setup() & loop()

Session 27 · 40 min
Goal: Install Arduino IDE + ESP32 support, upload Blink, explain setup() and loop().

Arduino IDE is the lowest-friction entry to embedded. Underneath, it's C++ with a thin framework. Every sketch has exactly two functions:

// runs once on power-up / reset
void setup() {
  pinMode(LED_BUILTIN, OUTPUT);
  Serial.begin(115200);
}

// runs forever after setup() returns
void loop() {
  digitalWrite(LED_BUILTIN, HIGH);
  delay(500);
  digitalWrite(LED_BUILTIN, LOW);
  delay(500);
}
Teach this way
  1. Install Arduino IDE 2.x together. Add ESP32 via Board Manager URL in Preferences.
  2. Plug in ESP32. Select Board + COM port.
  3. Examples → Basics → Blink. Upload. LED blinks.
  4. Change delay(1000) to delay(200). Upload. Direct cause-effect.
  5. Open Serial Monitor. Add Serial.println("Hello");. Magic moment.
  6. Save sketches in Git from day 1. Push Blink to GitHub now.
Common confusion
  • Wrong board selected → upload fails.
  • Wrong COM port. Windows may need CP210x driver.
  • Some ESP32 boards need BOOT held during upload. CP2102 DevKits usually auto-bootload.
  • Missing Serial.begin() → Serial Monitor shows nothing.
Verify understanding
  1. What runs first, setup() or loop()? setup()
  2. How many times does loop() run? Forever
  3. LED blink every 100 ms — what value? delay(100)
T3.6

GPIO — digitalRead / digitalWrite / pinMode

Session 28 · 45 min
Goal: Read a button, drive an LED, explain INPUT_PULLUP.

GPIO = General Purpose Input/Output. Three modes: OUTPUT, INPUT, INPUT_PULLUP.

INPUT_PULLUP enables an internal ~50k pull-up. Button wired between pin and GND: unpressed = HIGH, pressed = LOW (active-low).

const int BTN = 4;
const int LED = 2;

void setup() {
  pinMode(BTN, INPUT_PULLUP);
  pinMode(LED, OUTPUT);
}

void loop() {
  if (digitalRead(BTN) == LOW) digitalWrite(LED, HIGH);
  else digitalWrite(LED, LOW);
}
Teach this way
  1. Wire: LED on GPIO 2 + 330Ω; button between GPIO 4 and GND.
  2. Upload. Press button → LED on.
  3. Change to pinMode(BTN, INPUT) (no pull-up). Random behaviour — reinforces need.
  4. Toggle on press (not hold) → need debouncing + edge detection (later topic).
  5. ESP32 strapping pins: 0, 2, 12, 15 have boot functions. Warn.
Common confusion
  • Floating inputs. Tie somewhere always.
  • INPUT_PULLUP + button-to-GND → pressed = LOW. Counterintuitive.
  • GPIO current limit ~20 mA. Heavy loads need transistor.
  • Strapping pins as button inputs → board may not boot.
Verify understanding
  1. Mode for button-to-GND without external R? INPUT_PULLUP
  2. digitalRead on pressed INPUT_PULLUP pin? LOW
  3. Why avoid GPIO 12 on ESP32? Strapping pin — affects boot
T3.7

ADC — analogRead, Resolution, Reference

Session 29 · 40 min
Goal: Read a pot; convert raw counts to volts with V = raw · Vref / 2N.

ADC converts analog voltage to digital number. Two specs:

  • Resolution: Arduino Uno = 10-bit (0–1023), ESP32 = 12-bit (0–4095).
  • Reference: Uno = 5 V (Vcc), ESP32 = ~3.3 V (configurable attenuation).
V = raw · Vref / (2N − 1)
void setup() {
  Serial.begin(115200);
  analogReadResolution(12);
  analogSetAttenuation(ADC_11db);  // ~3.3V range on ESP32
}
void loop() {
  int raw = analogRead(34);
  float v = raw * 3.3f / 4095.0f;
  Serial.printf("raw=%d V=%.2f\n", raw, v);
  delay(200);
}
Teach this way
  1. Wire 10k pot: outer legs 3.3V/GND, wiper to GPIO 34.
  2. Upload. Turn pot, watch raw values 0→4095.
  3. She derives the formula: "3.3V full-scale, raw=2048, volts?" ~1.65V.
  4. ESP32 ADC quirks: nonlinear near rails, noisy (average 10 reads), ADC2 disabled during Wi-Fi.
  5. Demonstrate averaging: noise drops visibly.
Common confusion
  • 10-bit vs 12-bit resolution confusion.
  • Reading V > Vref → ADC clips. Use divider.
  • ADC2 during Wi-Fi → reads zero. Use ADC1 (GPIO 32–39).
  • Single reads: ~10–20 counts noise. Average.
Verify understanding
  1. 12-bit, Vref=3.3V. Raw=3000. V=? 2.42V
  2. Can ESP32 read 5V directly? No — use divider
  3. Why ADC2 fails during Wi-Fi? Shared hardware with RF
T3.8

PWM — Duty Cycle, Frequency, LEDC

Session 30 · 40 min
Goal: Dim an LED, drive motor speed with PWM; know LEDC on ESP32.

PWM = fast HIGH/LOW switching. Duty cycle = fraction HIGH. Average V = Vcc × duty. Frequency: >20 kHz for motors (no whine), >1 kHz for LEDs (no flicker).

Arduino: analogWrite(pin, 0–255). ESP32: use ledcSetup() + ledcWrite().

void setup() {
  ledcSetup(0, 5000, 8);        // ch 0, 5 kHz, 8-bit
  ledcAttachPin(2, 0);
}
void loop() {
  for (int d=0; d<=255; d++) { ledcWrite(0, d); delay(8); }
  for (int d=255; d>=0; d--) { ledcWrite(0, d); delay(8); }
}
Teach this way
  1. Wire LED with R on GPIO 2.
  2. Upload LEDC fade sketch. LED breathes.
  3. Scope the pin — show varying duty cycle.
  4. Change freq to 50 Hz → visible flicker. Back to 5k → smooth.
  5. Then drive MOSFET gate with PWM, motor on drain → variable speed.
Common confusion
  • PWM freq vs loop rate — different. PWM is hardware-timer.
  • Too-low freq → LED flicker, motor whine.
  • Duty 256 on 8-bit → overflow to 0.
  • PWM on pin conflicting with UART.
Verify understanding
  1. Duty=64 on 8-bit, 3.3V output. Avg V? 0.83V
  2. Why high freq for motors? Above audible, no whine
  3. Why PWM vs resistor for dimming LED? Efficient, wide range
T3.9

UART / Serial Communication

Session 31 · 40 min
Goal: Send text/binary between ESP32 and PC; debug silent-output.

Two wires (TX, RX) + GND. No shared clock. Both sides agree on baud rate (9600, 115200), 8N1.

ESP32 has 3 UARTs. UART0 is USB bridge + Serial Monitor.

void setup() {
  Serial.begin(115200);
  Serial.println("Ready!");
}
void loop() {
  if (Serial.available()) {
    String cmd = Serial.readStringUntil('\n');
    Serial.printf("You said: %s\n", cmd.c_str());
  }
}
Teach this way
  1. Upload echo. Serial Monitor at 115200, newline ending. Type, press Enter, echoed.
  2. Change Monitor to 9600 → gibberish. Baud matters.
  3. Wire ESP32 UART1 (17/16) to Arduino Uno UART (cross TX/RX, shared GND). Two MCUs talk.
  4. Show Serial2 usage on ESP32.
Common confusion
  • Baud mismatch.
  • Not crossing TX/RX.
  • Missing common GND.
  • 5V Arduino TX → 3.3V ESP32 RX (technically needs level shift or divider).
Verify understanding
  1. Different baud rates — what happens? Garbled / nothing
  2. Min wires between UART devices? 3 (TX, RX, GND)
  3. "8N1"? 8 data, No parity, 1 stop
T3.10

I2C — Wire Library, Addresses, Scanner

Session 32 · 50 min
Goal: Wire an I2C sensor (BMP280/OLED), scan for address, read data.

Two wires (SDA, SCL) + GND. Multiple devices share bus; each has a 7-bit address. Speeds: 100k/400k/1M Hz.

Pull-ups (4.7k to Vcc) required on SDA + SCL. Breakout boards usually include them.

#include <Wire.h>
void setup() {
  Serial.begin(115200); Wire.begin();
  for (byte a=1; a<127; a++) {
    Wire.beginTransmission(a);
    if (Wire.endTransmission()==0) Serial.printf("0x%02X\n", a);
  }
}
void loop() {}
Teach this way
  1. Wire BMP280: Vcc→3.3V, GND, SDA→GPIO 21, SCL→22.
  2. Run scanner. "Found: 0x76" — first sensor talk.
  3. Install Adafruit BMP280 library. Example → pressure/temp readings.
  4. Add OLED (0x3C). Scanner finds both. Shared-bus concept concrete.
  5. Scope SDA/SCL. Show bit patterns.
Common confusion
  • Missing pull-ups → no comms.
  • 3.3V / 5V mixing → need level shifter.
  • Address conflict → use ADDR pin.
  • Long wires >30cm → bus cap grows, comms fail.
Verify understanding
  1. Wires for I2C? 3: SDA, SCL, GND
  2. Two OLEDs same address? Conflict; use ADDR pin
  3. Pull-up value? 4.7k typ; 2.2k at higher speed
T3.11

SPI — Faster, More Wires, No Addresses

Session 33 · 40 min
Goal: Wire SPI device, know 4 lines, know when to pick SPI vs I2C.
SignalDirectionPurpose
MOSIMaster → SlaveData out from MCU
MISOSlave → MasterData in to MCU
SCKMaster → SlaveClock
CS / SSMaster → SlaveChip select, active low
#include <SPI.h>
const int CS = 5;
void setup() {
  SPI.begin();
  pinMode(CS, OUTPUT);
  digitalWrite(CS, HIGH);
}
byte readReg(byte r) {
  digitalWrite(CS, LOW);
  SPI.transfer(r | 0x80);
  byte v = SPI.transfer(0x00);
  digitalWrite(CS, HIGH);
  return v;
}
Teach this way
  1. Wire SPI OLED or SD card. 4 data + CS + Vcc/GND.
  2. Adafruit library example. File listing / display output.
  3. Scope SCK, MOSI — see clocked data. Compare to I2C last session.
  4. Tradeoff: SPI faster but more pins, one slave per CS line.
Common confusion
  • Wrong SPI mode (CPOL/CPHA) → shifted data.
  • Forgetting CS toggle.
  • Exceeding max SCK.
  • MOSI/MISO swap. "Master Out Slave In".
Verify understanding
  1. Wires for 1 SPI slave? 4 data + GND = 5
  2. 3 slaves, how many CS lines? 3
  3. Faster: SPI or I2C? SPI (tens of MHz vs ~1 MHz)
T3.12

Interrupts — attachInterrupt & ISR Rules

Session 34 · 40 min
Goal: Wire button to ISR; understand why ISRs stay short.

Interrupts let hardware fire code immediately. Modes: RISING, FALLING, CHANGE, LOW, HIGH.

ISR rules: Short (set flag, return); shared vars volatile; on ESP32 use IRAM_ATTR; debounce mechanical buttons.

volatile bool pressed = false;
void IRAM_ATTR onBtn() { pressed = true; }

void setup() {
  Serial.begin(115200);
  pinMode(4, INPUT_PULLUP);
  attachInterrupt(digitalPinToInterrupt(4), onBtn, FALLING);
}
void loop() {
  if (pressed) { Serial.println("!"); pressed = false; }
}
Teach this way
  1. Walk "set flag, handle in loop" pattern. Everyone wants to Serial.print in ISR and it breaks.
  2. Break rule: put Serial.println in ISR. Watch crash.
  3. Remove volatile — compiler may optimise out reads.
  4. Mechanical bounce → multiple triggers per press.
Common confusion
  • Long ops in ISR → hangs or watchdog.
  • Missing volatile → loop doesn't see changes.
  • Missing IRAM_ATTR on ESP32.
  • Button bounce not handled.
Verify understanding
  1. Why volatile for ISR vars? Don't cache; re-read every access
  2. What to NOT do in ISR? Serial prints, delays, I2C, long ops
  3. Mode for INPUT_PULLUP button press? FALLING
T3.13

Timing — delay(), millis(), Non-Blocking

Session 35 · 35 min
Goal: Write non-blocking multi-task loop using millis().

delay() blocks CPU. Fine for Blink alone; unusable for real code. Use millis() for non-blocking intervals.

unsigned long last = 0;
bool led = false;

void loop() {
  unsigned long now = millis();
  if (now - last >= 500) {   // non-blocking 500ms
    last = now; led = !led;
    digitalWrite(LED_BUILTIN, led);
  }
  if (digitalRead(4) == LOW) Serial.println("btn");
}
Teach this way
  1. Blocking version with delay(500) + button reads. Button presses missed during delay.
  2. Rewrite with millis() pattern. Button responsive.
  3. Rollover: millis rolls over ~49 days. (now - last >= interval) handles it in unsigned.
  4. Mention micros() for μs precision.
  5. Briefly mention FreeRTOS tasks on ESP32 — real answer for real multi-tasking. Not needed yet.
Common confusion
  • Signed int for millis → rollover breaks.
  • > vs >= — off-by-one.
  • Still using delay() for debouncing — OK for simple but use millis-based.
Verify understanding
  1. Why avoid delay() in multi-task? Blocks CPU — other tasks halt
  2. Why unsigned long? Correct rollover arithmetic
  3. μs-precision function? micros()
T3.14

Interfacing a New Sensor — Meta-Workflow

Session 36 · 60 min
Goal: Unfamiliar sensor + datasheet → wire + find library + read data in 30 min.

This is the meta-skill. Workflow:

  1. Datasheet scan: interface (I2C/SPI/analog)? Vcc? power? temp range?
  2. Pinout & wiring. Vcc, GND, data lines. Pull-ups on breakout?
  3. Find library. Adafruit/SparkFun/community typically have one.
  4. Use library example. Change pins, compile, upload.
  5. Verify Serial Monitor. Sane values? Calibration?
  6. Integrate: wrap in a helper function in your project.
Teach this way
  1. Give unfamiliar sensor (DS18B20 1-Wire temp, MPU6050 IMU).
  2. Don't help. Watch. Time her.
  3. She succeeds in 20 min or gets stuck. If stuck: diagnose the specific block.
  4. After success, she writes 3-line GitHub README: sensor, wiring, readings.
  5. Repeat weekly with new sensor. This muscle differentiates hardware engineers.
Common confusion
  • Wrong library choice — pick most-starred GitHub.
  • Missing I2C pull-ups if breakout omits them.
  • Unit confusion: °C vs °F, Pa vs hPa.
  • Too-fast sampling — MEMS need ms between reads.
Verify understanding
